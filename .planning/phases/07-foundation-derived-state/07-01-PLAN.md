    ```typescript
      // FOUND-03 D-08/D-09 — guarded ALTER for upgrade installs.
      // CREATE TABLE IF NOT EXISTS above handles fresh installs (local_date in DDL).
      // Existing v1.0 DBs need ALTER + backfill, which is not idempotent — guard it.
      const cols = await db.select<{ name: string }[]>("PRAGMA table_info(entries)");
      const hasLocalDate = cols.some((c) => c.name === "local_date");
      if (!hasLocalDate) {
        await db.execute("ALTER TABLE entries ADD COLUMN local_date TEXT");
        // D-09 — synchronous backfill from created_at (UTC day, best-effort per D-10).
        // Pre-migration entries near UTC midnight may be off by ±1 calendar day.
        await db.execute(
          "UPDATE entries SET local_date = strftime('%Y-%m-%d', created_at/1000, 'unixepoch') WHERE local_date IS NULL"
        );
        if (import.meta.env.DEV) {
          console.log("[db] Migrated entries.local_date column + backfilled existing rows");
        }
      }
    ```

    **Why exactly here:** The PRAGMA check must run AFTER the migration loop (so the entries table is guaranteed to exist on fresh installs — `CREATE TABLE IF NOT EXISTS` runs first) but BEFORE the dev diagnostic (so the table-list log reflects the post-migration state).

    **Why both DDL and ALTER:** `ALTER TABLE ADD COLUMN` is non-idempotent in SQLite (PITFALLS Anti-Pattern 6). Fresh installs hit the updated `CREATE TABLE` from Task 1 — the guard's `hasLocalDate` is `true`, the ALTER block is skipped. Upgrade installs from v1.0 see `hasLocalDate === false` on first run, ALTER fires, backfill populates rows, the column is now present; on second run the guard is `true`, block is skipped. Behavior is correct on both paths and re-running the app never crashes.

    **Why synchronous backfill:** D-09 / D-10 — app startup blocks until backfill completes (typical user DB <10k rows = <100ms). No async background job, no half-migrated state visible to widgets.

    **Why `WHERE local_date IS NULL`:** Defensive — if a partial migration left some rows already populated (e.g., user-debugged crash after ALTER but before UPDATE finished), this re-run only touches rows that still need it.

    **Do NOT** wrap in try/catch — if PRAGMA or ALTER fails, the app must surface the error at startup (consistent with the rest of `initializeDatabase` letting `db.execute` errors propagate).

    **Do NOT** add a `console.log` outside the `import.meta.env.DEV` guard — production users should not see migration chatter.
  </action>
