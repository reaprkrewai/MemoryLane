### Metadata Filtering Strategy

Per 06-RESEARCH.md, apply filters *before* vector similarity search:
1. Build SQL WHERE clause from date/tag/mood filters
2. Get filtered entry IDs (subset of all entries)
3. Vector search only within filtered entries (faster + reduces noise)

### Vector Normalization

Nomic-embed-text vectors need L2 normalization for cosine similarity:
```
magnitude = sqrt(sum(v_i^2))
normalized = [v_i / magnitude for each v_i]
```

Cosine similarity = dot product of normalized vectors.

  <action>
Create `src/utils/vectorSearchService.ts` with vector matching logic:

**Exports:**
1. `normalizeVector(vector: number[]): number[]`
   - Compute L2 norm: magnitude = sqrt(sum(v_i^2))
   - Return [v_i / magnitude for each element]
   - Handles zero-magnitude edge case (return as-is)

2. `cosineSimilarity(v1: number[], v2: number[]): number`
   - Assumes both vectors are normalized
   - Return: sum(v1[i] * v2[i])
   - Range: [-1, 1] (1 = identical direction, 0 = orthogonal, -1 = opposite)

3. `semanticSearch(query: string, options?: { dateRange?: { start: number; end: number }, tags?: string[], moods?: string[], limit?: number }): Promise<SearchEntry[]>`
   - Check Ollama health (via checkOllamaHealth from ollamaService)
   - If Ollama unavailable → fallback to keyword search from Phase 4 (return keywordSearch(query))
   - Generate query vector: queryEmbedding(query) from ollamaService
   - Normalize query vector
   - Build metadata filter SQL WHERE clause:
     ```
     WHERE e.created_at >= ? AND e.created_at < ? AND e.mood IN (...) AND e.id IN (
       SELECT entry_id FROM entry_tags WHERE tag_id IN (...)
     )
     ```
   - Query embeddings table:
     ```
     SELECT e.id, e.content, e.mood, e.word_count, e.created_at, e.updated_at, e.metadata,
            embeddings.vector
     FROM embeddings
     JOIN entries e ON embeddings.entry_id = e.id
     ${WHERE_CLAUSE}
     ORDER BY e.created_at DESC
     ```
   - In memory, compute cosine similarity for each returned row:
     - Deserialize vector BLOB as Float32Array
     - Normalize it
     - Compute dot product with normalized query vector
   - Sort by similarity descending
   - Return top K (default limit=10)
   - Return array of SearchEntry (same shape as Phase 4 keyword search results)

**Error handling:**
- If Ollama unavailable → return keywordSearch(query) (graceful fallback)
- If embedding query fails → throw error (caught by caller in SearchStore)
- If DB query fails → throw error

**Import:** getDb from lib/db, checkOllamaHealth + queryEmbedding from lib/ollamaService, keywordSearch from Phase 4 (will need to export from searchStore or create helper)

**Performance note:**
- Vector similarity search <100ms for typical journal (per 06-RESEARCH.md)
- Pre-filtering metadata before vector search reduces search space

## Notes

- **Float32Array binary storage:** Embeddings table stores vectors as BLOB (Float32Array buffer). Retrieval deserializes them for cosine similarity computation. Alternative: store as JSON, but binary is more efficient.
- **Metadata pre-filtering:** Filters applied in SQL WHERE clause *before* vector ranking reduces search space and improves latency.
- **Graceful fallback:** If Ollama unavailable at query time, semanticSearch() returns FTS5 keyword results instead. User is not blocked.
- **Query normalization:** Query vector normalized same as stored vectors for fair cosine similarity comparison.
