# Plan 06-03: Q&A Engine & Answer Grounding

## Objective

Enable users to ask natural language questions about their journal and receive AI-generated answers grounded in relevant entries. Implements the RAG (Retrieval-Augmented Generation) pattern: retrieve similar entries → augment LLM prompt → generate answer with citations.

**Purpose:** Move beyond search to understanding. Users ask "when did I last feel stressed?" and get: "You felt stressed about your project deadline in June (Entry X). You mentioned similar feelings in May (Entry Y)."

**Output:**
- Q&A service implementing RAG pattern
- Answer generation with [Entry ID] citations parsed and linked
- QA result display in SearchView with clickable citations
- Graceful handling if LLM model unavailable

## Context

### RAG Pipeline Overview

Per 06-RESEARCH.md Pattern 4:
1. **Retrieve:** Use semantic search (Plan 06-02) to find top K similar entries (K=5-10)
2. **Augment:** Extract full entry content + metadata, build context string
3. **Generate:** Send system prompt + context + user question to Ollama LLM
4. **Extract:** Parse response for [Entry XXXXX] citations, link back to entries
5. **Display:** Show answer + clickable citation list

### LLM Grounding

System prompt enforces grounding:
```
You are a helpful assistant answering questions about journal entries.
Always ground your answers in the provided entry content.
When you reference information from an entry, cite it using [Entry ID].
If you cannot answer from the entries, say so explicitly.
```

User prompt structure:
```
Based on these journal entries:

[Entry from YYYY-MM-DD]
{entry content}
[End Entry {entry_id}]

[Entry from YYYY-MM-DD]
{entry content}
[End Entry {entry_id}]

...

Question: {user_question}

Provide a grounded answer with citations.
```

LLM output expected to include: [Entry xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx] format citations.

### LLM Model Management

- **Default model:** llama2:7b (balanced speed + quality)
- **Alternative:** llama2:13b (user can configure in settings, Phase 6 defers settings UI to Phase 06-04)
- **Model detection:** Health check already knows if LLM available (LLMAI.llm true/false from aiStore)

Create `src/utils/qaService.ts` implementing the RAG pattern:

**Exports:**

1. `parseCitations(text: string): string[]`
   - Regex to extract [Entry XXXXX] patterns from LLM response
   - Pattern: /\[Entry ([a-f0-9-]+)\]/g (matches UUID format)
   - Returns array of unique entry IDs (deduped)
   - Example: text = "You felt stressed [Entry abc123]. Similar in [Entry def456]."
   - Returns: ["abc123", "def456"]

2. `askQuestion(question: string, retrievedEntries: Array<{ id: string; content: string; created_at: number }>): Promise<{ answer: string; citedEntryIds: string[] }>`
   - Retrieves top K similar entries (reuses semanticSearch from Plan 06-02, K=5-10)
   - Builds context string:
     ```typescript
     const context = retrievedEntries
       .map((e) => {
         const date = new Date(e.created_at).toLocaleDateString();
         return `[Entry from ${date}]\n${e.content}\n[End Entry ${e.id}]`;
       })
       .join("\n\n");
     ```
   - Constructs system + user prompts (as described above)
   - Checks LLM availability: `const health = await checkOllamaHealth()`
   - If LLM unavailable: Return { answer: "LLM model not available. Install llama2:7b via Ollama.", citedEntryIds: [] }
   - Calls `askQuestion()` from ollamaService (note: same function name, different module — clarify with import alias)
   - Parses response: `const citedEntryIds = parseCitations(rawAnswer)`
   - Returns { answer: rawAnswer, citedEntryIds }
   - On error: throw Error with description (caught by caller in SearchStore)

**Imports:**
- getDb from lib/db
- semanticSearch from utils/vectorSearchService
- checkOllamaHealth, askQuestion as askQuestionOllama from lib/ollamaService (use alias to avoid naming conflict)

**Temperature tuning:**
- Temperature 0.7 (balanced factuality + creativity)
- top_p: 0.9 (nucleus sampling for diversity)
- These are hardcoded in the askQuestion call to ollamaService (Plan 06-01 task 1 defines these)

**Performance notes:**
- Retrieval: <100ms (semantic search, Plan 06-02)
- LLM inference: 5-8 seconds typical for 7b model
- Total Q&A latency: <10s per 06-RESEARCH.md

## Notes

- **Citation format:** [Entry XXXXX] in LLM response must match the entry ID format. Ensure regex in parseCitations() matches UUID pattern.
- **Context length:** Sending ~5-10 full entries to LLM may exceed token limits for very long entries. Consider truncating to 500-char summary if needed (defer to v2).
- **Hallucination prevention:** System prompt emphasizes grounding. If LLM still hallucinates (invents info not in entries), consider stricter prompt or temperature reduction to 0.5 (but may reduce helpfulness).
- **Multi-turn conversation:** v1 is single-query Q&A. Multi-turn (follow-up questions) deferred to v2.
- **Answer preservation:** When user switches tabs or searches again, Q&A result clears (can be made sticky in v2).
